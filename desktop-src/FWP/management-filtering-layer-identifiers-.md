# <a name="filtering-layer-identifiers"></a>Filtrage des identificateurs de couche

Les identificateurs de couche WFP (Windows Filtering Platform) sont tous représentés par un GUID. Ces identificateurs sont définis comme suit.

Les suffixes V4 et V6 à la fin des identificateurs de couche indiquent si la couche se trouve dans la pile réseau IPv4 ou dans la pile réseau IPv6.

<dl> <dt>

<span id="FWPM_LAYER_INBOUND_IPPACKET_V4___FWPM_LAYER_INBOUND_IPPACKET_V6"></span><span id="fwpm_layer_inbound_ippacket_v4___fwpm_layer_inbound_ippacket_v6"></span>**Couche FWPM entrée de la couche \_ \_ \_ IPPACKET \_ v4/FWPM \_ \_ entrante \_ IPPACKET \_ V6**
</dt> <dd> <dl> <dt>



Cette couche de filtrage se trouve dans le chemin de réception juste après l’analyse de l’en-tête IP d’un paquet reçu, mais avant l’exécution de tout traitement d’en-tête IP. Aucun déchiffrement ou réassemblage IPsec n’a eu lieu.


</dt> </dl> </dd> <dt>

<span id="FWPM_LAYER_INBOUND_IPPACKET_V4_DISCARD___FWPM_LAYER_INBOUND_IPPACKET_V6_DISCARD"></span><span id="fwpm_layer_inbound_ippacket_v4_discard___fwpm_layer_inbound_ippacket_v6_discard"></span>**Couche \_ FWPM \_ entrante \_ IPPACKET \_ v4 \_ ignorée/FWPM \_ couche \_ entrante \_ IPPACKET \_ V6 \_ ignorée**
</dt> <dd> <dl> <dt>



Cette couche de filtrage se trouve dans le chemin de réception pour inspecter tous les paquets reçus qui ont été ignorés au niveau de la couche réseau.


</dt> </dl> </dd> <dt>

<span id="FWPM_LAYER_OUTBOUND_IPPACKET_V4___FWPM_LAYER_OUTBOUND_IPPACKET_V6"></span><span id="fwpm_layer_outbound_ippacket_v4___fwpm_layer_outbound_ippacket_v6"></span>**Couche \_ FWPM \_ sortante \_ IPPACKET \_ v4/FWPM \_ \_ sortante \_ IPPACKET \_ V6**
</dt> <dd> <dl> <dt>



Cette couche de filtrage se trouve dans le chemin d’envoi juste avant l’évaluation de la fragmentation du paquet envoyé. Tout le traitement de l’en-tête IP est terminé et tous les en-têtes d’extension sont en place. L’authentification et le chiffrement IPsec ont déjà eu lieu.


</dt> </dl> </dd> <dt>

<span id="FWPM_LAYER_OUTBOUND_IPPACKET_V4_DISCARD___FWPM_LAYER_OUTBOUND_IPPACKET_V6_DISCARD"></span><span id="fwpm_layer_outbound_ippacket_v4_discard___fwpm_layer_outbound_ippacket_v6_discard"></span>**Couche \_ FWPM \_ sortante \_ IPPACKET \_ v4 \_ ignorée/ \_ FWPM \_ sortie \_ IPPACKET \_ V6 \_ ignorée**
</dt> <dd> <dl> <dt>



Cette couche de filtrage se trouve dans le chemin d’envoi pour inspecter tous les paquets envoyés qui ont été ignorés au niveau de la couche réseau.


</dt> </dl> </dd> <dt>

<span id="FWPM_LAYER_IPFORWARD_V4___FWPM_LAYER_IPFORWARD_V6"></span><span id="fwpm_layer_ipforward_v4___fwpm_layer_ipforward_v6"></span>**Couche \_ FWPM \_ IPFORWARD \_ v4/FWPM \_ \_ IPFORWARD \_ V6**

</dt> <dd> <dl> <dt>

Cette couche de filtrage est située en haut de la couche réseau et non au bas de la couche de transport, de sorte que tous les paquets envoyés par des transports tiers ou en tant que paquets bruts sont filtrés au niveau de cette couche.

## <a name="remarks"></a>Notes

Ces identificateurs de couche de filtrage sont également appelés identificateurs de couche de filtrage de gestion. L’API WFP contient également un ensemble d' [identificateurs de couche de filtrage d’exécution](/windows-hardware/drivers/network/management-filtering-layer-identifiers), documentés dans le kit WDK (Windows Driver Kit). Les identificateurs de couche de filtrage au moment de l’exécution sont LUID et, par conséquent, sont plus petits, seulement 64 bits en taille, par rapport aux identificateurs de couche de filtrage de gestion, qui sont de 128 bits.

Les identificateurs de couche et de filtrage d’exécution du filtrage de gestion pointent vers les mêmes couches.

Les identificateurs de couche de filtrage de gestion sont utilisés par les fonctions qui interagissent avec le moteur de filtrage de base (BFE) à partir du mode utilisateur ou du mode noyau (par exemple, [**FwpmFilterAdd0**](/windows/desktop/api/Fwpmu/nf-fwpmu-fwpmfilteradd0)).

Les identificateurs de couche de filtrage au moment de l’exécution sont utilisés par les fonctions qui interagissent avec le moteur de filtre en mode noyau uniquement (par exemple, [FwpsFlowAssociateContext0](/windows-hardware/drivers/ddi/fwpsk/nf-fwpsk-fwpsflowassociatecontext0), [FwpsStreamInjectAsync0](/windows-hardware/drivers/ddi/fwpsk/nf-fwpsk-fwpsstreaminjectasync0)) et dans les structures de données qui proviennent directement du noyau (par exemple, [**FWPM \_ NET \_ Event \_ classifi \_ DROP0**](/windows/desktop/api/Fwpmtypes/ns-fwpmtypes-fwpm_net_event_classify_drop0)).
