# <a name="m-isolated-applications-and-side-by-side-assemblies"></a>M (applications isolées et assemblys côte à côte)

[A](a-sbscs-gly.md) B C [D](d-sbscs-gly.md) e F [G](g-sbscs-gly.md) H [I](i-sbscs-gly.md) J K L M N O [P](p-sbscs-gly.md) Q [R](r-sbscs-gly.md) [S](s-sbscs-gly.md) T U V W X Y Z

<dl> <dt>

<span id="_win32_manifest_file_gly"></span><span id="_WIN32_MANIFEST_FILE_GLY"></span>**fichier manifeste**
</dt> <dd>

Fichier XML qui décrit un assembly côte à côte ou une application isolée. Un fichier manifeste peut contenir des métadonnées de liaison et d’activation traditionnellement stockées dans le registre, telles que les classes COM, les interfaces et les bibliothèques de types. Chaque application isolée ou assembly côte à côte est accompagné d’un fichier manifeste. Un manifeste d’application décrit une application totalement ou partiellement isolée et un manifeste d’assembly décrit un assembly côte à côte.

</dd> <dt>

<span id="_win32_metadata_gly"></span><span id="_WIN32_METADATA_GLY"></span>**Metadata**
</dt> <dd>

Informations stockées dans un fichier manifeste à propos de la liaison ou de l’activation d’une application isolée ou d’un assembly côte à côte. Par exemple, les informations sur les classes COM, les interfaces et les bibliothèques de types, traditionnellement stockées dans le registre, peuvent être stockées sous forme de métadonnées dans un manifeste. Les métadonnées décrivent les éléments qui composent un assembly et ses dépendances vis-à-vis d’autres assemblys côte à côte. Les métadonnées spécifiées dans le manifeste de l’application et les assemblys dépendants peuvent être utilisés pour remplir le contexte d’activation.

</dd> </dl>
