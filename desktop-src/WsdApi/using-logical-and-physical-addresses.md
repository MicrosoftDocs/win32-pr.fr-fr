<span data-ttu-id="e1a4f-104">[WS-Discovery](https://specs.xmlsoap.org/ws/2005/04/discovery/ws-discovery.pdf) définit l’adressage logique à l’aide d’URI basés sur le `urn:uuid:` format.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-104">[WS-Discovery](https://specs.xmlsoap.org/ws/2005/04/discovery/ws-discovery.pdf) defines logical addressing using URIs based on the `urn:uuid:` format.</span></span> <span data-ttu-id="e1a4f-105">L’objectif de ce schéma d’adressage est de différencier l’identité de l’appareil de son adresse IP actuelle.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-105">The purpose of this addressing scheme is to differentiate the identity of the device from its current IP address.</span></span> <span data-ttu-id="e1a4f-106">Ce schéma fournit essentiellement les fonctionnalités des noms DNS sans nécessiter de serveur de noms.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-106">This scheme essentially provides the functionality of DNS names without requiring a name server.</span></span> <span data-ttu-id="e1a4f-107">[Devices Profile for Web services](https://specs.xmlsoap.org/ws/2006/02/devprof/) (DPWS) recommande que les appareils utilisent ce schéma d’adressage.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-107">[Devices Profile for Web Services](https://specs.xmlsoap.org/ws/2006/02/devprof/) (DPWS) recommends that devices use this addressing scheme.</span></span>

<span data-ttu-id="e1a4f-108">DPWS recommande également que les services utilisent des adresses physiques (également appelées transport).</span><span class="sxs-lookup"><span data-stu-id="e1a4f-108">DPWS also recommends that services use physical (also called transport) addresses.</span></span> <span data-ttu-id="e1a4f-109">Cela permet aux clients qui ne prennent pas en charge de manière native WS-Discovery mécanismes d’adressage de communiquer avec les services DPWS.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-109">This enables clients that do not natively support WS-Discovery addressing mechanisms to communicate with DPWS services.</span></span> <span data-ttu-id="e1a4f-110">En outre, chaque service peut définir ses adresses, ce qui permet l’adressage au niveau du transport pour les implémentations d’appareils qui gèrent la répartition de service sur une couche inférieure.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-110">Also, each service can define its addresses, which enables transport level addressing for device implementations that manage service dispatch at a lower layer.</span></span> <span data-ttu-id="e1a4f-111">Enfin, l’utilisation d’adresses physiques optimise l’interopérabilité.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-111">Finally, using physical addresses maximizes interoperability.</span></span>

<span data-ttu-id="e1a4f-112">L’inconvénient de l’adressage physique est qu’il complique les implémentations d’appareils, car l’adresse IP ou de transport actuelle doit faire l’objet d’un suivi et les métadonnées de l’appareil doivent être modifiées en conséquence.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-112">The disadvantage of physical addressing is that it adds complexity to device implementations, as the current IP or transport address must be tracked and device metadata must be modified accordingly.</span></span> <span data-ttu-id="e1a4f-113">Pour cette raison, DPWS ne requiert pas de services pour utiliser des adresses de transport.</span><span class="sxs-lookup"><span data-stu-id="e1a4f-113">For this reason, DPWS does not require services to use transport addresses.</span></span>
